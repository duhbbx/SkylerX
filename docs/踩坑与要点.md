# 踩坑与要点

本项目的踩坑与解法增量沉淀。只记要点，按主题归类，方便速查。

## 达梦 DM

- **对象类型在 `all_objects` 里是 `object_type='CLASS'`，不是 Oracle 的 `'TYPE'`。** 列对象类型、查类型 DDL 都要用 `CLASS`：列表 `WHERE object_type='CLASS'`；`dbms_metadata.get_ddl('CLASS', …)`（用 `'TYPE'` 报 -20008），类型体是 `'CLASS_BODY'`（带空格的 `'CLASS BODY'` 报 -20001）。其余对象类型（TABLE/VIEW/FUNCTION/PROCEDURE/PACKAGE/SEQUENCE/TRIGGER/SYNONYM）与 Oracle 一致。见 `core-driver/dialects/dm.ts`、`ui/ddl.ts`。
- **`all_tables.dropped` 在 DM 是 `NULL`（无回收站），不是 Oracle 的 `'YES'/'NO'`。** 过滤回收站对象要用 `dropped IS NULL OR dropped <> 'YES'`，直接 `dropped='NO'` 会把 DM 的表全滤光。见 `ui/migrate/introspect.ts` 的 `readOracle`（Oracle/DM 共用）。

## SQL 执行

- **「An object could not be cloned」最常见的真凶：把 Vue 响应式 Proxy 传进 `ipcRenderer.invoke`。** 网格提交 `doCommit` 把 `editPreview.value.stmts`（`ref` 里的对象 → 深层响应式 Proxy 数组）直接丢给 `executeBatch` 的 IPC invoke，Electron 结构化克隆拒绝 Proxy → 报这句通用错，且是在**渲染层出站时**抛的，根本没到主进程（所以在主进程 catch 里包 `unwrapAggregate` 没用）。`execute()` 不中招是因为它收的是纯字符串 SQL。修法：出站前拍平成纯数据——调用点 `[...p.stmts]`，并在 preload `executeBatch` 兜底 `Array.from(statements, String)` + `{...options}`。**任何传给 IPC 的参数都要确保是纯对象/数组，不能带 reactive/ref Proxy。**
- 次要相关：pg 在连接失败 / 多错误时抛 `AggregateError`（带 `errors` 数组），跨 IPC 也克隆不了。`execute()` 用 `unwrapAggregate()` 压成普通 `Error(message)`；`executeBatch` / `beginSession` / `executeInSession` / `commit/rollbackSession` 也补上了（防主进程侧抛非可克隆错盖掉真因）。见 `core-driver/dialects/postgres.ts`。

- **语句拆分器必须识别 PL/SQL 块，不能按裸分号切。** `BEGIN … NULL; END;` 里的分号是块内分隔，按 `;` 拆会把触发器/函数/过程/包体从中间切断（达梦/Oracle 报 -2007 语法错）。`ui/sqlSplit.ts` 的规则：`CREATE [OR REPLACE] TRIGGER/FUNCTION/PROCEDURE/PACKAGE/TYPE`、`DECLARE`、匿名 `BEGIN` 块内的 `;` 不拆，由单独一行的 `/`（SQL*Plus 风格）或输入结束终止；`BEGIN;`/`BEGIN WORK` 等事务控制不算 PL/SQL；PG 的 `$$ … $$` 函数体里的 `;` 也要保护。

## 跨方言类型渲染（已实测稳）

- **DBX(Tauri) 那批类型显示 bug，SkylerX 没有**（用成熟驱动 + `fmt` 对对象走 `JSON.stringify`）：实测 Vastbase `int2`→number、`jsonb`→object（渲染成 JSON 串非 `[object Object]`）、`uuid`→string、复合类型→`(s,c)` 串（非 null）；MySQL `tinyint(1)`→number（**没**被强转布尔）。回归用例见 `docs/qa/features/type-rendering.md`。
- **剪贴板导入**：`parseCSV(text, delim)` 加了分隔符参数，ImportDialog「粘贴剪贴板」按第一行有无 `\t` 自动判 TSV(电子表格)/CSV。`io.test.ts` 有 TSV 回归。
- **IPv6 host 不踩 DBX 的坑**：连接表单 host/port 是分开字段，不解析 `host:port`，所以 `::1` 直接传给驱动，不会报 invalid port。
- **SQL 拆分早就处理字符串里的分号**（Navicat 痛点）：`sqlSplit` 跳过字符串/反引号/双引号/注释/PLSQL 块，`'a;b'` 不会被当终止符（`sqlSplit.test.ts` 有覆盖）。

## 跨方言 introspection（reader）

- **`information_schema.COLUMNS` / `all_tab_columns` 不区分基表与视图。** 直接按列建表会把视图也当成表。各 reader 要先取基表名再按基表过滤：MySQL/SQLServer 用 `TABLE_TYPE='BASE TABLE'`，Oracle/DM 用 `all_tables`。只有 PG reader 因为用 `relkind IN ('r','p')` 天生免疫。
- **MySQL 约束名只在表内唯一**（每个表主键都叫 `PRIMARY`，FK 名也可跨表重名）。按约束聚合的 Map 要用 `表+约束名` 复合键，否则两表同名约束会串列（如主键变成 `['id','id']`）。

## 主题 / Monaco

- **主题支持「跟随系统」**：`settings.theme` 现为 `'dark'|'light'|'system'`，默认 `system`（老用户保留已存值）。`settings.ts` 用 `window.matchMedia('(prefers-color-scheme: dark)')` 解析，并监听 OS 切换；导出响应式 `resolvedTheme`（解析后的 dark/light）。`data-theme` 属性据此切换 CSS 变量。
- **Monaco 编辑器配色要跟随主题**：之前 `SqlEditor.vue` 硬编码 `theme:'vs-dark'`，浅色模式下编辑器还是黑的。改为 watch `resolvedTheme` 调 `monaco.editor.setTheme('vs'|'vs-dark')`（全局 API，一次切全部实例）。全项目只有 `SqlEditor.vue` 调 `monaco.editor.create`。

## AI provider

- **本地 provider（Ollama）不需要 API Key。** Ollama 暴露 OpenAI 兼容端点 `http://localhost:11434/v1`，走现有 `callOpenAiCompat` 即可。新增 `AiProvider='ollama'`、`LOCAL_AI_PROVIDERS` 集合、`isLocalAiProvider()`。请求时 `resolveKey()` 给本地 provider 一个占位 Bearer（Ollama 忽略它），其余 provider 仍强制要 key。
- **嵌入端点也支持本地免 key（离线 RAG）**：`canEmbed()` / `embedTexts()` 用 `isLocalEmbeddingBase(base)`（匹配 localhost / 127.0.0.1 / ::1 + 可选端口）判断，本地端点空 key 给占位 Bearer。设置里有「用 Ollama 本地」一键预设（baseUrl=11434、model=nomic-embed-text、清空 key）。这样 RAG / 向量记忆能全程离线。
- **所有 AI 入口的「没填 key」前置拦截要统一用 `isActiveAiConfigured()`**，别再各自 `aiProviders[provider].apiKey.trim()`，否则本地模型会被误拦。已改：AiAssistant / AiHealthCheck / MockData / AiComment / SqlTranslate / AiChatPanel（含 provider 下拉的可见性过滤）。本地 provider 判「已配置」只看 baseUrl。
- **主进程 `ai:fetch` 用 Node 全局 fetch(undici)，默认不读 HTTP_PROXY**，所以能直连 localhost:11434，不受本机 Privoxy 影响（那条只管开发者的 curl）。
- **AI 流式输出**：主进程 `ai:stream` 用 `res.body.getReader()` 读流，把 **raw SSE 文本块**经 `webContents.send('ai:stream:<reqId>', {chunk})` 推回；**SSE 帧解析 + provider 差异留在渲染层** `ai.ts` 的 `aiHttpStream`（按空行切帧、取 `data:` 行、`[DONE]` 停，跨 chunk 累积 buf）。`askAiChatStream` 解析 OpenAI 的 `choices[0].delta.content` / Anthropic 的 `content_block_delta.delta.text`。preload 用 `ipcRenderer.on(channel, listener)` + invoke 的 `.finally(removeListener)` 收尾。流式超时放到 120s（本地模型冷启动慢）。
- **流式有「空回退」**：某些代理忽略 `stream:true` 直接返回整包，流里拿不到 token → `AiChatPanel.send()` 检测 `reply` 为空就再走一发非流式 `askAiChat`。
- **改响应式数组元素要取数组里的代理**：往 `messages.value` push 普通对象后，必须 `const slot = messages.value[len-1]` 再改 `slot.content`，直接改 push 进去的原始对象引用不会触发重渲染。

## 定时任务 / 调度作业

- **DB 侧定时任务管理**（`ScheduledJobsDialog.vue`）按 `familyOf` 分四套：MySQL `information_schema.EVENTS`（启停 `ALTER EVENT … ENABLE/DISABLE`、`DROP EVENT`，需 `event_scheduler=ON`）、PG `cron.job`（pg_cron 扩展，`cron.alter_job(id, active:=)` / `cron.unschedule(id)`）、Oracle+DM `all_scheduler_jobs`（`DBMS_SCHEDULER.ENABLE/DISABLE/DROP_JOB`）、SQL Server `msdb..sysjobs`（`sp_update_job` / `sp_delete_job`，需 SQL Agent）。每方言一个 Adapter（listSql/label/enabled/en/disable/dropSql/template/hint），表格通用渲染 + 动作列。自带连接选择，入口 `act:jobs`（⌘K + 数据库菜单）。
- **已存在、勿重复造**：执行计划可视化（`plan.ts` + `PlanPanel.vue`，含成本热力/ANALYZE 耗时/倾斜）、提交前 DML 预览（QueryPane `editPreview` 模态）、外键跟随（QueryPane `onFkNavigate`）都已实现。本轮只新增了结果网格右键菜单（按值过滤/复制 SQL/复制 IN/跟随外键）+ 选中行 Calc 汇总，以及定时任务管理。

## 执行计划热力树 —— 多方言

- **热力树原先只有 PG**（`parsePgPlan`），MySQL 只给 `FORMAT=TREE` 文本。现扩到：MySQL 默认 `EXPLAIN FORMAT=JSON` → `parseMysqlPlan`（递归识别 `nested_loop[]`/`table`/`ordering_operation` 等不规则算子键，cost 取 `cost_info.prefix_cost`/`query_cost`）；analyze 仍走 `EXPLAIN ANALYZE` 树形文本。Oracle/DM 用两步 `EXPLAIN PLAN FOR` + 读 `PLAN_TABLE`，按 `id/parent_id` 建树（`parseOraclePlan`，列名大小写不敏感）。
- **Oracle 两步必须同连接**：`PLAN_TABLE` 是会话级 GTT，`prep`(EXPLAIN PLAN) 和读取若落在不同池连接上读不到。所以 `planQuery` 返回带 `prep` 时，QueryPane 走 `beginSession`→两次 `executeInSession`→`endSession`，保证同一连接。
- **统一出口** `buildPlanData(format, result)`：把一次 EXPLAIN 的 result 转成 `{tree, text}`（pg-json/mysql-json 取首格 JSON、oracle-rows 取整行、mysql-tree 取文本），QueryPane 不再自己抠首格。`PlanPanel` 复用同一 `PlanNode`，无需改。
- 注：OceanBase / GBase8a 归在 mysql 家族也会走 `FORMAT=JSON`，个别版本若不支持会报错回退（普通表格 EXPLAIN）。

## 命令面板 / 对话框连接选择

- **「需要先选连接」的全局功能改成对话框内自带连接选择**，不再为每条连接在 ⌘K 里塞一条（连接一多 palette 就刷屏）。已改：服务器活动 / Schema 快照 / 备份还原（ServerMonitor 早先已是这个模式）。做法：dialog 自己 `conns = ref([])`、`connId = ref('')`、`conn = computed(...)`，`onMounted` 拉 `client.connections.list()`，顶部一个 `<select v-model="connId">`，body `v-if="conn"` 门控，`watch(connId)` 重载。
- **Workspace 侧**：palette 单条 `act:activity` / `act:snapshots` / `act:backup`，open state 退成 bool；菜单命令经 `PALETTE_KEY_MAP` 直接映射到这些 id（不再走「打开 palette 让用户挑」的早返回分支）。
- **TS narrow 注意**：`conn` 是 computed，`if (!conn.value) return` 之后在循环里再读 `conn.value.x` 不会被收窄；要 `const c = conn.value` 存下来用。函数内的 `cols.map((c)=>…)` 等局部参数别再叫 `c`，会跟连接 `c` 撞（已改名 `col`）。

## Notebook

- **Notebook 从弹框改成常驻 tab**：新增 `NotebookPane.vue` + `QueryTabs` 的 `kind:'notebook'`。`Tab.conn` 因此改成可选（笔记本不绑单一连接，面板内自带连接选择）；所有 `t.conn.id` 等访问按 kind 守卫加 `?.`/`!`（模板里 query/mongo/redis 等分支用 `t.conn!`，tab 条的 DialectIcon 对 notebook 用 📓 兜底）。`openNotebook()` 单例（已开则聚焦），不进「重开关闭标签」栈。
- **运行 cell 报「No database selected」根因**：notebook 只选了连接、没把库传给 execute。MySQL 连接无默认库时裸 SQL 必失败。修法：面板内加「库/schema」级联选择（同 ConnTargetPicker：顶层 metadata 分 Database/Schema），`execute(connId, sql, [], { database, schema, maxRows })` 把库带上（跟 QueryPane 的 `execOptions()` 一致）。库/schema 持久化进 Notebook 模型。
- **「保存不生效」其实是弹框被销毁**：旧 `NotebookDialog` 是 `v-if` 挂载，关掉即销毁、重开 `blank()`，存的笔记本只在「打开…」下拉里，给人「没保存」错觉。改 tab 后常驻；并在 `onMounted` 自动恢复最近一次编辑的笔记本（`listNotebooks()[0]`），保存按钮加 toast 反馈。

## 标签页 / 快捷键

- **菜单 accelerator 与渲染层 keydown 不要双注册同一快捷键**：⌘/Ctrl+Shift+T「重开关闭标签」只在主进程 `menu.ts` 注册 accelerator → `send('reopen-tab')`，渲染层 `Workspace.onKeydown` 不再处理，否则 Electron 菜单和 window keydown 都触发会一次重开两个。⌘W 关标签同理（菜单专属）。
- **重开关闭的标签**靠 `QueryTabs` 里的 `closedStack`（去 id、上限 10）。query tab 在 `close()` 时通过 QueryPane 暴露的 `getSql()` 把当前 SQL 存进 `draft`，重开经 `:initial-sql` 填回（只填不执行）。NoSQL（mongo/redis/es）tab 不入栈。
